# v2.3 — Trading Safety & Execution Quality

## Context

v2.2 addresses configuration hygiene (per-account params/fees, connections management). v2.3 targets the gaps that directly affect capital protection and trading accuracy — identified during architecture review.

These are the changes that move the needle on live trading outcomes, not code quality.

---

## 1. Hard Circuit Breaker on Calculator

**Problem:** When weekly loss or drawdown hits the hard stop (`"limit"` state), the calculator still runs and returns a position size with `eligible: False`. The trader sees the number and can ignore the red flag.

**Fix:** The `/calculator/calculate` endpoint must **refuse to calculate** when limits are breached.

**Files:**
- `api/routes_calculator.py` — add guard before `run_risk_calculator()`
- `core/state.py` — `portfolio_stats.weekly_pnl_state`, `portfolio_stats.dd_state`

**Behavior:**
```
if weekly_pnl_state == "limit" or dd_state == "limit":
    → return error fragment: "BLOCKED: [Weekly loss / Drawdown] limit reached"
    → do NOT run risk_engine
    → do NOT log to pre_trade_log
    → show which limit was hit and current value vs threshold
```

**Edge cases:**
- If engine restarts and hasn't received enough data to calculate drawdown yet, default to "allow" (not enough data to block)
- If both limits are hit, show both
- Calculator should also show a warning (not block) when state is `"warning"`

## 2. Estimated vs Actual Slippage Tracking

**Problem:** `risk_engine.py:132-162` estimates VWAP slippage from orderbook snapshot. The estimate is never compared to actual fill price. Over 100+ trades, systematic bias is invisible.

**Fix:** When a fill arrives (via WS or Quantower bridge), match it to the most recent `pre_trade_log` entry for the same symbol and compute the delta.

**Files:**
- `core/handlers.py` or `core/reconciler.py` — add matching logic
- `core/database.py` / `core/db_trades.py` — add `slippage_actual` column to `execution_log`
- `api/routes_analytics.py` — surface aggregate slippage stats

**Flow:**
```
1. Fill event arrives (symbol=BTCUSDT, fill_price=67,432.50, qty=0.5)
2. Query pre_trade_log: most recent calc for BTCUSDT where qty ≈ fill qty
3. Extract est_fill_price from that calc
4. Compute: actual_slippage = (fill_price - est_fill_price) / est_fill_price
5. Store in execution_log.slippage_actual
6. Over time: avg(slippage_actual - slippage_estimated) = estimator bias
```

**Matching heuristic:**
- Same symbol, within last 5 minutes, closest quantity match
- If no match found, store `slippage_actual = NULL` (manual/external trade)

**Analytics output:**
- Mean slippage estimation error (bps)
- Slippage by exchange, by time of day, by volatility regime
- Flag if estimator is consistently underestimating (sizing too aggressively)

## 3. Stale-Data Guard on Calculator

**Problem:** If WS drops and REST fallback is on a 30s cycle, equity/positions can be 30 seconds stale. The calculator uses stale unrealized PnL → wrong available equity → wrong position size.

**Fix:** Calculator checks data freshness before running.

**Files:**
- `api/routes_calculator.py` — add freshness check
- `core/state.py` — expose `last_data_update_ts` (already tracked in `ws_status`)

**Behavior:**
```
data_age = now - last_ws_or_rest_update

if data_age > 120s:
    → BLOCK calculation
    → "Position data is stale (last update {data_age}s ago). Refresh required."
    → offer manual refresh button (triggers REST fetch)

if data_age > 30s:
    → WARN banner: "Data may be stale ({data_age}s). Consider refreshing."
    → allow calculation but flag in pre_trade_log
```

**The refresh button** triggers `exchange.py:fetch_positions()` + `fetch_account()` on demand, then re-renders the calculator.

## 4. Exchange Contract Spec Validation

**Problem:** Calculator can output position sizes that violate exchange minimums (min notional, min qty, lot step). A 0.0001 BTC position on Binance might be below min notional. The trader discovers this only when the broker rejects the order.

**Fix:** Validate calculated size against `exchange_info` contract specs before returning.

**Files:**
- `core/risk_engine.py` — add post-sizing validation in `run_risk_calculator()`
- `core/exchange.py` — expose min notional, lot step, tick size from cached `exchange_info`

**Checks:**
```python
# After position size is calculated:
min_notional = exchange_info[symbol]["min_notional"]   # e.g., 5 USDT
lot_step     = exchange_info[symbol]["lot_step"]        # e.g., 0.001
min_qty      = exchange_info[symbol]["min_qty"]          # e.g., 0.001

# Round size to lot_step
final_size = round_to_step(raw_size, lot_step)

# Check minimums
if final_size * entry_price < min_notional:
    result["eligible"] = False
    result["reject_reason"] = f"Below min notional ({min_notional} USDT)"

if final_size < min_qty:
    result["eligible"] = False
    result["reject_reason"] = f"Below min qty ({min_qty})"
```

**Per-exchange nuances:**
- Binance: `filters` array in exchange_info contains `MIN_NOTIONAL`, `LOT_SIZE`, `PRICE_FILTER`
- Bybit/OKX: different filter names but CCXT normalizes most of them via `market["limits"]`
- Store parsed limits in a lookup dict on `fetch_exchange_info()`, keyed by symbol

## 5. Position-Exchange Reconciliation on Startup

**Problem:** On restart, engine loads last DB equity snapshot (could be hours old), then fetches live positions. During the gap between snapshot restore and first WS update, risk calculations use stale equity.

**Fix:** Force a full REST reconciliation before allowing calculator to run after startup.

**Files:**
- `main.py` — lifespan startup sequence
- `core/state.py` — `is_initializing` flag (already exists)

**Flow:**
```
Startup:
1. Load DB snapshot (equity, bod_equity, etc.) — existing
2. Fetch live positions via REST — existing
3. Fetch live account balance via REST — existing
4. Compare DB equity vs live equity
5. If delta > 1%: log WARNING with exact delta
6. Update app_state with live values
7. THEN set is_initializing = False (allow calculator)
```

**Key change:** Steps 2-6 must complete BEFORE `is_initializing` flips to False. Currently `_reinit()` is launched as a background task — the calculator could be hit before reconciliation finishes.

## Implementation Order

1. **Item 1** (Circuit breaker) — smallest change, highest capital protection impact
2. **Item 3** (Stale-data guard) — small change, prevents wrong sizing
3. **Item 4** (Contract spec validation) — moderate change, prevents rejected orders
4. **Item 5** (Startup reconciliation) — moderate change, prevents stale-equity sizing
5. **Item 2** (Slippage tracking) — larger change, improves sizing accuracy over time
