## Problem

The engine has no order lifecycle tracking. TP/SL is fetched via REST polling (slow, doesn't clear on cancel). Position closes are detected by heuristic. Fills map trade side to position direction incorrectly. Two positions on the same symbol collide everywhere.

## Identity Model — Dual-ID Design

Every object (order, fill, position) exists on two systems: the **exchange** (Binance, Bybit, etc.) and the **terminal** (Quantower, Sierra Chart, NinjaTrader, etc.). Each assigns its own ID.

| Column | Source | Purpose | Example |
|--------|--------|---------|---------|
| `exchange_order_id` | Exchange (Binance `orderId`) | REST reconciliation, stable across sessions | `"184729345"` |
| `terminal_order_id` | Terminal (Quantower `Order.UniqueId`) | Real-time matching during session | `"qt_a3f8c201"` |

**How Quantower exposes both:**
- `Order.Id` = the exchange's native order ID (Binance `orderId`) → `exchange_order_id`
- `Order.UniqueId` = Quantower's internal unique key → `terminal_order_id`
- Same pattern for `Trade.Id`/`Trade.UniqueId` and `Position.Id`/`Position.UniqueId`

**When only a terminal is connected (no direct exchange REST):**
- Both columns are still populated — the terminal passes through exchange IDs from the broker connection.
- If the terminal doesn't expose the exchange ID (e.g. paper trading), `exchange_order_id` is NULL and we fall back to `terminal_order_id` for all matching.

**UNIQUE constraint:** `(account_id, exchange_order_id)` — uses NULL (not empty string) when exchange ID is unavailable. SQLite allows multiple NULLs in UNIQUE columns, so paper-trading orders won't conflict.

**Lookup priority:**
1. `exchange_order_id` (stable, survives restarts, matches REST data)
2. `terminal_order_id` (session-scoped, real-time matching)
3. `(symbol, direction)` (last resort fallback)

## Order State Machine

```
new ──► partially_filled ──► filled
 │          │
 ├──► canceled
 ├──► expired
 └──► rejected

partially_filled ──► canceled (remaining qty canceled)
```

- **No `pending` state** — we don't send orders from the engine. Orders arrive from the terminal already exchange-acknowledged.
- **Terminal states** (no transitions out): `filled`, `canceled`, `expired`, `rejected`
- **Active states** (visible in Open Orders): `new`, `partially_filled`
- **Invalid transitions are rejected** — if a snapshot tries `filled → new`, log warning and skip. Prevents data corruption.

**Terminal status mapping:**
```
Quantower "Opened"          → new
Quantower "PartiallyFilled" → partially_filled
Quantower "Filled"          → filled
Quantower "Cancelled"       → canceled
Quantower "Refused"         → rejected
Quantower "Inactive"        → expired
```

Implementation: `core/order_state.py` (NEW) — `OrderStatus` enum + `VALID_TRANSITIONS` dict + `validate_transition()`.

## PositionInfo Addition

One new field on the in-memory `PositionInfo` dataclass (`core/state.py`):

```python
individual_fees: float = 0.0  # cumulative fees, computed from fills DB (never accumulated in cache)
```

Derived fields (computed in template/API, not stored):
- `gross_pct = individual_unrealized / (average * contract_amount) * 100`
- `net_pnl = individual_unrealized - individual_fees`
- `net_pct = net_pnl / (average * contract_amount) * 100`

**Fees are always computed via `SELECT SUM(fee) FROM fills WHERE terminal_position_id = ?`**, never via `+= fee` accumulation. This guarantees idempotency on duplicate fills and correctness after restart.

## Architecture — OrderManager

All order domain logic lives in `OrderManager` (`core/order_manager.py` — NEW), decoupled from WebSocket transport:

```python
class OrderManager:
    """Domain logic for order lifecycle. No WS/HTTP knowledge."""

    def __init__(self, db: DatabaseManager):
        self._db = db
        self._open_orders: list[dict] = []   # cached for dashboard

    async def process_order_snapshot(self, account_id: int, orders: list[dict]) -> None:
        """Upsert orders, detect cancellations, enrich positions with TP/SL."""

    async def process_fill(self, account_id: int, fill: dict) -> None:
        """Record fill, update parent order, refresh position fees from DB."""

    async def on_position_closed(self, prev: PositionInfo) -> None:
        """Build closed_positions row from fills. Called 2s after position disappears."""

    def enrich_positions_tpsl(self, positions: list) -> None:
        """Set TP/SL on positions from cached open orders.
        When multiple TP/SL exist, pick the one closest to mark price."""

    @property
    def open_orders(self) -> list[dict]:
        return self._open_orders
```

**Instantiation:** Singleton on `PlatformBridge.__init__()`:
```python
class PlatformBridge:
    def __init__(self):
        self._order_manager = OrderManager(db)
```

`platform_bridge` parses messages and delegates to `OrderManager`. This makes the logic testable, reusable for future terminals, and keeps transport concerns separate.

## DB vs Cache Architecture

### Storage Rules

| Data | Primary Store | Cache | Why |
|------|--------------|-------|-----|
| Open positions | Cache (`app_state.positions`) | — | Real-time, changes every tick |
| Open orders | DB (`orders` table) | `OrderManager._open_orders` | Dashboard needs fast read, DB is truth |
| TP/SL on positions | DB (`orders` WHERE type=tp/sl) | Copied to `positions[].individual_tp/sl_price` | Dashboard reads cache |
| Position fees | DB (`fills` SUM fee) | Copied to `positions[].individual_fees` | Computed from DB, never accumulated |
| Fills | DB (`fills` table) | No cache | Read on-demand for history tab |
| Order history | DB (`orders` table) | No cache | Read on-demand for history tab |
| Closed positions | DB (`closed_positions` table) | No cache | Read on-demand for history tab |

### Critical Rule: DB-First, Cache-Second

```python
# CORRECT: DB is source of truth
await db.upsert_order_batch(orders)                        # 1. DB write
self._open_orders = await db.query_open_orders(account_id)  # 2. Cache FROM DB
self.enrich_positions_tpsl(app_state.positions)              # 3. Enrich cache

# WRONG: cache-first risks divergence on DB failure
self._open_orders = orders             # ← if DB write below fails, cache is wrong
await db.upsert_order_batch(orders)
```

### TP/SL Source — Orders Table Only

TP/SL on positions comes **exclusively** from the `orders` table via `enrich_positions_tpsl()`. Position snapshots do **NOT** preserve TP/SL from `prev`. This eliminates the stale-TP-on-cancel bug entirely:

- Order placed → order_snapshot → orders table has TP/SL → enriched on positions → dashboard shows it
- Order canceled → order_snapshot → orders table no longer has it → positions cleared → dashboard clears
- No more REST polling for TP/SL

## Write Path — Event Handlers

### Order Snapshot Processing

```python
async def process_order_snapshot(self, account_id: int, orders: list[dict]) -> None:
    # 1. Fetch all current active orders in ONE query (not per-order)
    existing = await self._db.get_active_orders_map(account_id)

    # 2. Validate state transitions in memory (fast)
    valid_orders = []
    for o in orders:
        eid = o["exchange_order_id"]
        if eid in existing:
            if not validate_transition(existing[eid]["status"], o["status"]):
                log.warning("Invalid transition %s→%s for order %s", 
                           existing[eid]["status"], o["status"], eid)
                continue
        valid_orders.append(o)

    # 3. Batch upsert in single transaction (1 query)
    await self._db.upsert_order_batch(valid_orders)

    # 4. Mark active orders NOT in snapshot as canceled (1 query)
    active_ids = [o["exchange_order_id"] for o in orders if o["exchange_order_id"]]
    await self._db.mark_stale_orders_canceled(account_id, active_ids)

    # 5. Rebuild cache from DB (1 query)
    self._open_orders = await self._db.query_open_orders_all(account_id)

    # 6. Enrich positions with TP/SL (closest to mark price)
    self.enrich_positions_tpsl(app_state.positions)
```
**Total: 4 queries** regardless of order count (not N+1).

**TP/SL selection (closest to mark price):**
```python
def enrich_positions_tpsl(self, positions):
    for pos in positions:
        mark = pos.fair_price or pos.average
        # Find all active TP/SL orders for this position
        tp_orders = [o for o in self._open_orders
                     if o["symbol"] == pos.ticker
                     and o["position_side"] == pos.direction
                     and o["order_type"] in ("take_profit", "take_profit_market")
                     and o["status"] in ("new", "partially_filled")]
        sl_orders = [o for o in self._open_orders
                     if o["symbol"] == pos.ticker
                     and o["position_side"] == pos.direction
                     and o["order_type"] in ("stop_loss", "stop_market")
                     and o["status"] in ("new", "partially_filled")]

        # Closest to mark price = triggers next
        if tp_orders:
            pos.individual_tp_price = min(tp_orders, key=lambda o: abs(o["stop_price"] - mark))["stop_price"]
        else:
            pos.individual_tp_price = 0.0
        if sl_orders:
            pos.individual_sl_price = min(sl_orders, key=lambda o: abs(o["stop_price"] - mark))["stop_price"]
        else:
            pos.individual_sl_price = 0.0
        pos.individual_tpsl = pos.individual_tp_price > 0 or pos.individual_sl_price > 0
```

### Fill Processing

```python
async def process_fill(self, account_id: int, fill: dict) -> None:
    # 1. Upsert fill to DB (dedup by exchange_fill_id)
    await self._db.upsert_fill(fill)

    # 2. Update parent order (best-effort — order may not exist yet)
    if fill.get("exchange_order_id"):
        try:
            await self._db.update_order_from_fill(fill["exchange_order_id"], fill)
        except Exception:
            log.warning("Parent order %s not found for fill", fill["exchange_order_id"])

    # 3. Refresh position fees from DB (SUM query, not accumulate)
    pos_id = fill.get("terminal_position_id", "")
    if pos_id:
        fees = await self._db.get_position_fees(account_id, pos_id)
        for pos in app_state.positions:
            if pos.position_id == pos_id:
                pos.individual_fees = fees
                break
```

### Position Close Detection — Partial + Full

Closing fills trigger `closed_positions` rows **immediately** (not just on full position disappearance). Each closing fill (or group of fills from the same order) creates one row.

**Two triggers:**
1. **Closing fill arrives** → `process_fill()` detects `is_close=true` → builds row for the closed portion
2. **Position fully disappears** → safety net: check for any unrecorded closing fills, build remaining rows

```python
# Trigger 1: In process_fill(), after DB write:
if fill["is_close"]:
    # Defer 2s to allow related fills (partial fills from same order) to arrive
    asyncio.get_event_loop().call_later(
        2.0,
        lambda: asyncio.ensure_future(
            self._build_close_row_for_fill(account_id, fill)
        )
    )

# Trigger 2: In _handle_position_snapshot, when position disappears:
for key in old_keys - new_keys:
    prev_position = existing_by_key[key]
    asyncio.get_event_loop().call_later(
        2.0,
        lambda p=prev_position: asyncio.ensure_future(
            self._build_final_close_row(p)
        )
    )
```

```python
async def _build_close_row_for_fill(self, account_id: int, fill: dict) -> None:
    """Build a closed_positions row for a partial or full close."""
    pos_id   = fill["terminal_position_id"]
    symbol   = fill["symbol"]
    direction = fill["direction"]
    close_qty = fill["quantity"]

    # Get opening fills for this position to compute VWAP entry
    opens = await self._db.get_position_fills(
        account_id, pos_id, symbol, direction, is_close=False
    )
    if not opens:
        # Fallback: use position's average price from cache
        pos = next((p for p in app_state.positions if p.position_id == pos_id), None)
        entry_price = pos.average if pos else 0.0
        entry_time  = 0
    else:
        entry_price = sum(f["price"] * f["quantity"] for f in opens) / sum(f["quantity"] for f in opens)
        entry_time  = min(f["timestamp_ms"] for f in opens)

    # Get all closing fills from the same order (partial fills grouped)
    close_fills = await self._db.get_fills_by_order(
        account_id, fill["exchange_order_id"]
    ) if fill.get("exchange_order_id") else [fill]
    close_fills = [f for f in close_fills if f["is_close"]]

    exit_price   = sum(f["price"] * f["quantity"] for f in close_fills) / sum(f["quantity"] for f in close_fills)
    exit_time    = max(f["timestamp_ms"] for f in close_fills)
    closed_qty   = sum(f["quantity"] for f in close_fills)
    realized_pnl = sum(f["realized_pnl"] for f in close_fills)
    total_fees   = sum(f["fee"] for f in close_fills)
    # Include proportional entry fees: (closed_qty / total_position_qty) * total_entry_fees
    entry_fees   = sum(f["fee"] for f in opens)
    total_open_qty = sum(f["quantity"] for f in opens) if opens else closed_qty
    total_fees  += entry_fees * (closed_qty / total_open_qty) if total_open_qty else 0

    # Exit reason from parent order type
    exit_reason = await self._determine_exit_reason(fill.get("exchange_order_id", ""))

    # Implementation shortfall
    shortfall = await self._compute_shortfall(symbol, direction, entry_price, exit_price, entry_time)

    await self._db.insert_closed_position({
        "terminal_position_id": pos_id,
        "symbol":        symbol,
        "direction":     direction,
        "quantity":       closed_qty,
        "entry_price":   entry_price,
        "exit_price":    exit_price,
        "entry_time_ms": entry_time,
        "exit_time_ms":  exit_time,
        "realized_pnl":  realized_pnl,
        "total_fees":    total_fees,
        "net_pnl":       realized_pnl - total_fees,
        "hold_time_ms":  exit_time - entry_time if entry_time else 0,
        "exit_reason":   exit_reason,
        **shortfall,
    })
    await event_bus.publish("risk:position_closed", {"symbol": symbol, "direction": direction})


async def _build_final_close_row(self, prev: PositionInfo) -> None:
    """Safety net: when position fully disappears, check for unrecorded closing fills."""
    # Query all closing fills for this position that don't have a closed_positions row yet
    unrecorded = await self._db.get_unrecorded_closing_fills(
        app_state.active_account_id, prev.position_id, prev.ticker, prev.direction
    )
    if unrecorded:
        # Group by exchange_order_id and build rows for each
        for order_id, fills in group_by_order(unrecorded).items():
            await self._build_close_row_for_fill(
                app_state.active_account_id, fills[0]
            )
```

## Race Conditions & Mitigations

| Race | Scenario | Mitigation |
|------|----------|------------|
| Fill before order | Fill references order not yet in DB | `upsert_fill()` succeeds independently. `update_order_from_fill()` is best-effort. Next order snapshot reconciles |
| Rapid snapshots | Two snapshots 10ms apart | 500ms throttle in plugin. Engine processes under `app_state._lock` serially |
| Position gone before fill | PositionRemoved fires before TradeAdded | 2s deferred close — allows pending fills to arrive |
| Partial close double-row | Same closing order has multiple partial fills | Group fills by exchange_order_id, single row per order |
| Cache divergence | DB write fails, cache already updated | DB-first always. Cache is rebuilt FROM DB after write |
| Duplicate fills | Same fill from historical backfill + live | UNIQUE on `exchange_fill_id` — upsert deduplicates. Fees via SUM, never accumulate |
| Plugin disconnect | Orders stay "new" forever | `last_seen_ms` on orders. Background staleness loop marks as canceled after 5 min |
| Position flip | LONG 10 → SELL 20 = SHORT 10 | Fills correctly tag `is_close` + `direction` via `PositionImpactType`. PositionCloser handles LONG close naturally |

## Staleness Detection

Background task in `schedulers.py`:
```python
async def _order_staleness_loop():
    """Every 60s: check for active orders not seen in 5+ minutes."""
    while True:
        await asyncio.sleep(60)
        if not platform_bridge.is_connected:
            continue
        stale = await db.mark_stale_orders(
            account_id=app_state.active_account_id,
            stale_threshold_ms=5 * 60 * 1000,
        )
        if stale:
            log.warning("Marked %d stale orders as canceled", stale)
```

`mark_stale_orders` scoped to ACTIVE states only:
```sql
UPDATE orders SET status='canceled', updated_at_ms=?
WHERE account_id=? AND status IN ('new','partially_filled')
  AND last_seen_ms < ? AND last_seen_ms > 0
```

### 3. `OnOrderChanged` sends full snapshot
```csharp
private void OnOrderChanged(Order order)
{
    if (_conn is null) return;
    long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
    if ((now - _lastOrderSignalMs) < OrderSignalMinIntervalMs) return;  // 500ms throttle
    _lastOrderSignalMs = now;

    var account = _activeAccount ?? Core.Instance.Accounts.FirstOrDefault();
    if (account is null) return;
    var orders = (Core.Instance.Orders ?? Enumerable.Empty<Order>())
        .Where(o => o.Account?.Id == account.Id)
        .ToArray();
    var snap = RiskEngineEventMapper.MapOrders(account, orders);
    _ = _conn.SendOrderSnapshotAsync(snap);
}

## Adapter Layer — Broker-Agnostic Protocols

The engine never touches raw exchange data. All exchange/terminal data flows through normalized protocols defined in `core/adapters/protocols.py`. Each adapter (Binance, Bybit, future exchanges) maps raw responses to these types. The `OrderManager` only consumes normalized data.

```
                    ┌──────────────────┐
                    │   OrderManager   │  ← broker-agnostic, only sees normalized types
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │  Binance   │  │   Bybit    │  │  Future    │   ← each maps raw → normalized
     │  Adapter   │  │  Adapter   │  │  Adapter   │
     └────────────┘  └────────────┘  └────────────┘
              │              │              │
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ Binance API│  │ Bybit API  │  │    ???     │   ← raw exchange responses
     └────────────┘  └────────────┘  └────────────┘

     ┌─────────────────────────────────────────────┐
     │           Terminal Plugin (C#)              │
     │  Quantower / Sierra / NinjaTrader           │   ← also maps to same normalized types

## Idempotency Guarantees

| Operation | Guarantee | How |
|-----------|-----------|-----|
| Order upsert | Idempotent | UNIQUE on exchange_order_id, INSERT OR REPLACE |
| Fill upsert | Idempotent | UNIQUE on exchange_fill_id, INSERT OR REPLACE |
| Closed position | No duplicates | UNIQUE on (position_id, exit_time_ms) |
| Fee computation | Always correct | `SUM(fee)` from DB, never `+= fee` |
| TP/SL enrichment | Always current | Rebuilt from orders table on every snapshot |

### Phase 6: OrderManager — Core Logic (engine-side)

**6A. OrderManager skeleton**
- Create `core/order_manager.py`
- Constructor takes `DatabaseManager`
- Empty method stubs for `process_order_snapshot`, `process_fill`, `on_position_closed`, `enrich_positions_tpsl`
- `_open_orders` cache list

**6B. process_order_snapshot()**
- Parse orders, map status via `QT_STATUS_MAP`
- Fetch active orders map (1 query)
- Validate transitions in memory
- Batch upsert (1 transaction)
- Mark missing as canceled (1 query, scoped to ACTIVE_STATES)
- Rebuild `_open_orders` cache from DB
- Call `enrich_positions_tpsl()`

**6C. enrich_positions_tpsl()**
- From `_open_orders` cache, find TP/SL orders per (symbol, position_side)
- When multiple: pick closest to mark price
- Set `individual_tp_price`, `individual_sl_price`, `individual_tpsl` on positions
- Clear to 0 when no matching orders (cancel clears immediately)

**6D. process_fill()**
- Upsert fill to DB
- Update parent order (best-effort)
- Refresh `individual_fees` from DB via `get_position_fees()` (SUM, not accumulate)
- If `is_close`: schedule `_build_close_row_for_fill()` with 2s delay

### Phase 7: Position Closer

**7A. PositionCloser — closing fill trigger**
- `_build_close_row_for_fill()` in OrderManager
- Query opening fills → VWAP entry price
- Group closing fills by order → VWAP exit price
- Compute fees (proportional entry fees + closing fees)
- Determine `exit_reason` from parent order type
- Insert `closed_positions` row
- Publish `risk:position_closed` event

**7B. PositionCloser — position disappearance safety net**
- `_build_final_close_row()` in OrderManager
- Query unrecorded closing fills
- Build remaining rows for any fills not yet captured in 7A
- Called 2s after position disappears from snapshot

**7C. Implementation shortfall**
- `_compute_shortfall()` in OrderManager
- Query `pre_trade_log` for matching (symbol, direction) within time window before entry
- Compare intended entry/exit vs actual
- Compute bps difference

| Backfill depth? | **Last 30 days** from exchange_history |
