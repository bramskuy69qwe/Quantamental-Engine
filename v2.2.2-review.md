# v2.2.2 — Architectural Review & Maximized Execution Plan

---

## Part 1: Critical Review of Initial Plan

### 1. State Management — Gaps Found

**Missing states:**
- `pending` — order submitted to exchange but no ACK yet (network in-flight)
- `rejected` — exchange refused the order (insufficient margin, invalid price, self-trade prevention)
- `amended` — order quantity/price modified (Binance supports this via `PUT /fapi/v1/order`)

**No explicit state machine.** The plan lists statuses but doesn't define valid transitions. Without this, invalid transitions (e.g., `filled → new`) can corrupt data silently.

Valid state machine should be:
```
pending ──► new ──► partially_filled ──► filled
  │          │          │                   
  │          ├──► canceled               
  │          ├──► expired                
  │          └──► rejected               
  └──► rejected                          
                                          
partially_filled ──► canceled (remaining qty canceled)
```

**Order amendments not handled.** When a user moves a limit price or changes quantity on a working order, the plan has no mechanism to detect this. Quantower fires `OrderUpdated` (not just Added/Removed), but the plugin only subscribes to Added/Removed. The amendment would be captured by the next order snapshot, but the `orders` table has no `previous_price` or amendment history.

**Recommendation:** Add `Order.Updated` subscription in plugin. Add `amendment_count` column to orders table. Consider an `order_events` audit log table for full state history (low priority but valuable for debugging).

### 2. Race Conditions & Concurrency — Gaps Found

**Race 1: Fill arrives before order snapshot.**
A fill event references `exchange_order_id = "123"`, but no order with that ID exists in the DB yet (the order snapshot hasn't arrived or was delayed). The `update_order_from_fill()` call would silently fail or error.

**Mitigation:** `upsert_fill()` should succeed independently. `update_order_from_fill()` should be best-effort (log warning if order not found). The next order snapshot will reconcile.

**Race 2: Two order snapshots in rapid succession.**
`OnOrderChanged` fires for both `OrderAdded` and `OrderRemoved` within milliseconds. Two snapshots are sent. If processed concurrently, `mark_missing_orders_canceled()` from snapshot A could cancel orders that are valid in snapshot B.

**Mitigation:** Process order snapshots serially under `app_state._lock`. The 500ms throttle in the plugin helps but doesn't eliminate this. Add a sequence number to snapshots and only process if `seq > last_processed_seq`.

**Race 3: Position snapshot vs fill timing.**
Fill arrives (closing fill) → engine writes to fills table → position snapshot arrives showing position gone → `PositionCloser` queries fills → the fill is there. This is the happy path. But if position snapshot arrives BEFORE the fill (Quantower fires PositionRemoved before TradeAdded), `PositionCloser` can't find the closing fill and builds an incomplete `closed_positions` row.

**Mitigation:** `PositionCloser` should be deferred — schedule it with a 2-3s delay after position disappearance to allow pending fills to arrive. If no closing fill found after delay, log a warning and build a best-effort row from the last known position state.

**Race 4: Cache divergence from DB.**
If `db.upsert_order()` fails (disk full, SQLite lock), but the cache (`app_state.open_orders`) is already updated, the dashboard shows stale data. On next order snapshot, the DB gets the correct state, but the failed write is silently lost.

**Mitigation:** Write DB first, then update cache. If DB write fails, don't update cache — log error and let the next snapshot retry. The cache should always be a projection of DB state.

**Race 5: Concurrent SQLite writes.**
The engine is single-threaded (asyncio) but multiple coroutines can await DB writes interleaved. SQLite WAL mode helps with concurrent reads, but writes are still serialized by SQLite's lock. Multiple rapid upserts from order snapshot + fill + position snapshot could contend.

**Mitigation:** Batch order upserts into a single transaction (one `executemany` instead of N individual inserts). This is both faster and avoids lock contention.

### 3. Architecture & Integration — Gaps Found

**No interface for order lifecycle.** The plan puts all logic directly in `platform_bridge.py`, mixing transport (WebSocket handling) with domain logic (order state management). This makes it hard to test and hard to add a second terminal (Sierra).

**Recommendation:** Extract an `OrderManager` class that owns:
- Order state machine validation
- DB persistence
- Cache enrichment
- TP/SL resolution

`platform_bridge` just parses the message and delegates to `OrderManager`. Same for any future Sierra/Ninja adapter.

**PositionCloser coupling.** It's triggered inline from `_handle_position_snapshot()`. If it fails (DB error, missing fills), it blocks position snapshot processing.

**Recommendation:** Fire-and-forget via event bus: `await event_bus.publish("risk:position_closed", {...})`. Handler runs independently, can retry on failure.

**No abstraction for terminal-agnostic order mapping.** Currently `_map_fill()` in platform_bridge.py has Quantower-specific field names. When Sierra is added, a second mapper would be needed.

**Recommendation:** Define a `TerminalAdapter` protocol with `map_order()`, `map_fill()`, `map_position()` methods. Each terminal plugin has its own adapter. Low priority — can be done when second terminal is added.

### 4. Data Consistency — Gaps Found

**Cache invalidation on TP/SL.** When an order snapshot arrives, TP/SL is enriched on `app_state.positions`. But `_handle_position_snapshot()` (which fires more frequently than order snapshots) rebuilds positions from scratch. The current code preserves TP/SL via `prev`, but if `prev` is None (first snapshot after restart), TP/SL is lost until the next order snapshot.

**Mitigation:** On engine restart, process order snapshot BEFORE position snapshot. The plugin should send order snapshot first in `ConnectAndIntroduceAsync()`. Add a flag `_orders_initialized` — if position snapshot arrives before orders, defer TP/SL enrichment.

**Fee accumulation drift.** `individual_fees` is accumulated in cache via `pos.individual_fees += fill["fee"]`. If a fill is replayed (duplicate), fees double-count. If engine restarts, fees reset to 0 until `populate_open_position_metadata()` runs.

**Mitigation:** Never accumulate in cache. Always compute from DB: `SELECT SUM(fee) FROM fills WHERE terminal_position_id = ?`. Cache this value and refresh on each fill. Slightly more DB reads but guarantees correctness.

**`mark_missing_orders_canceled` is dangerous.** If the plugin sends a snapshot that only includes working orders (status=Opened), all filled/canceled orders in DB that aren't in the snapshot would be incorrectly re-canceled. Need to scope: only mark as canceled orders that were previously `new` or `partially_filled` in DB and are missing from the active snapshot.

**Mitigation:** `mark_missing_orders_canceled` should be: `UPDATE orders SET status='canceled', updated_at_ms=? WHERE account_id=? AND status IN ('new','partially_filled') AND exchange_order_id NOT IN (?)`.

### 5. Error Handling & Edge Cases — Gaps Found

**Plugin disconnect mid-lifecycle.** Orders in DB stay as `new`/`partially_filled` forever. No staleness detection.

**Mitigation:** Add `last_seen_ms` column to orders. On each order snapshot, update `last_seen_ms` for all active orders. Background task: if `last_seen_ms` is >5 minutes old and status is still active, mark as `stale` or trigger a REST reconciliation.

**Duplicate fill handling.** Historical backfill sends fills with `qt:{trade_id}` keys. Live fills arrive with the same `trade_id`. Without deduplication, fees double-count and PnL is wrong.

**Mitigation:** The UNIQUE constraint on `(account_id, exchange_fill_id)` handles this — upsert will update, not duplicate. But `terminal_fill_id` might differ between historical and live for the same fill. Use `exchange_fill_id` as the canonical dedup key.

**Partial position close.** User closes 50% of a position via a market order. Position still exists (smaller qty) but fills table has a closing fill. `PositionCloser` shouldn't fire because the position didn't disappear.

**Mitigation:** `PositionCloser` only fires when position is fully gone from snapshot. Partial closes are recorded as fills with `is_close=true` but don't create a `closed_positions` row. The `closed_positions` row is only built when the position fully disappears. At that point, ALL fills (including partial closes) are aggregated.

**Position flip.** User is LONG 10, places a SELL 20 — result is SHORT 10. Quantower may fire: PositionRemoved (LONG) + PositionAdded (SHORT) in quick succession. The LONG position's `PositionCloser` needs to handle this correctly — the closing fill's `is_close=true` portion is 10 qty, and the remaining 10 opens the SHORT.

**Mitigation:** Position flips are handled naturally if fills correctly tag `is_close` and `direction`. The LONG `PositionCloser` will find the closing fills (10 qty) and build the row. The SHORT position starts fresh. Quantower's `PositionImpactType` handles this (`Reverse` type).

**Exchange rejects order.** User places an order, exchange returns `REJECTED`. If the plugin sends the order in the snapshot before rejection, it appears in `orders` table as `new`. The rejection arrives later and status should update to `rejected`.

**Mitigation:** The next order snapshot either won't contain the rejected order (if Quantower removes it) or will contain it with status `Refused`. `mark_missing_orders_canceled` handles the first case. Status mapping handles the second (map Quantower `Refused` → engine `rejected`).

### Phase 1: Architecture & Design Pattern Setup

**1.1 Order State Machine** (`core/order_state.py` — NEW)
```python
from enum import Enum
from typing import Set

class OrderStatus(str, Enum):
    PENDING          = "pending"
    NEW              = "new"
    PARTIALLY_FILLED = "partially_filled"
    FILLED           = "filled"
    CANCELED         = "canceled"
    EXPIRED          = "expired"
    REJECTED         = "rejected"

# Valid transitions — any transition not in this map is illegal
VALID_TRANSITIONS: dict[OrderStatus, Set[OrderStatus]] = {
    OrderStatus.PENDING:          {OrderStatus.NEW, OrderStatus.REJECTED},
    OrderStatus.NEW:              {OrderStatus.PARTIALLY_FILLED, OrderStatus.FILLED, 
                                   OrderStatus.CANCELED, OrderStatus.EXPIRED, OrderStatus.REJECTED},
    OrderStatus.PARTIALLY_FILLED: {OrderStatus.FILLED, OrderStatus.CANCELED},
    # Terminal states — no transitions out
    OrderStatus.FILLED:           set(),
    OrderStatus.CANCELED:         set(),
    OrderStatus.EXPIRED:          set(),
    OrderStatus.REJECTED:         set(),
}

TERMINAL_STATES = {OrderStatus.FILLED, OrderStatus.CANCELED, OrderStatus.EXPIRED, OrderStatus.REJECTED}
ACTIVE_STATES   = {OrderStatus.PENDING, OrderStatus.NEW, OrderStatus.PARTIALLY_FILLED}

def validate_transition(current: str, target: str) -> bool:
    try:
        return OrderStatus(target) in VALID_TRANSITIONS[OrderStatus(current)]
    except (ValueError, KeyError):
        return False
```

### Phase 3: State Management & Open Order Lifecycle

**3.1 Order Snapshot Processing** (in OrderManager)
```
1. Parse all orders from snapshot
2. Map terminal status → OrderStatus enum (validate)
3. For each order:
   a. Check if exists in DB (by exchange_order_id)
   b. If exists: validate state transition (current → new)
   c. If invalid transition: log warning, skip (don't overwrite filled with new)
   d. If valid or new: upsert
4. Batch upsert in single transaction
5. Mark active orders NOT in snapshot as canceled (scoped to ACTIVE_STATES only)
6. Update last_seen_ms for all orders in snapshot
7. Rebuild open_orders cache from DB query
8. Enrich positions with TP/SL from open orders
```

**3.2 Fill Processing** (in OrderManager)
```
1. Parse fill, assign direction from plugin (not inferred from side)
2. Upsert fill to DB (dedup by exchange_fill_id)
3. Find parent order by exchange_order_id:
   a. If found: update filled_qty, avg_fill_price, status
   b. If not found: log warning (order snapshot may arrive later)
4. Refresh position fees from DB (SELECT SUM, not accumulate)
5. If is_close: no immediate action (wait for position to disappear)
```

**3.3 Position Close Detection** (deferred)
```
1. Position disappears from snapshot
2. Schedule PositionCloser with 2s delay (allow pending fills to arrive)
3. After delay:
   a. Query fills for this position (by terminal_position_id, fallback symbol+direction)
   b. Separate opening fills (is_close=0) from closing fills (is_close=1)
   c. If no closing fills found: log warning, build best-effort row from position state
   d. Compute VWAP entry/exit, fees, hold_time
   e. Determine exit_reason from closing fill's parent order type
   f. Match to pre_trade_log for implementation shortfall
   g. Insert closed_positions row
   h. Publish "risk:position_closed" event
   i. Reconciler picks up for async MFE/MAE computation
```

**3.4 Staleness Detection** (background task in schedulers.py)
```python
async def _order_staleness_loop():
    """Every 60s: check for orders that haven't been seen in 5+ minutes."""
    while True:
        await asyncio.sleep(60)
        if not platform_bridge.is_connected:
            continue
        stale_count = await db.mark_stale_orders(
            account_id=app_state.active_account_id,
            stale_threshold_ms=5 * 60 * 1000,
        )
        if stale_count:
            log.warning("Marked %d stale orders as canceled", stale_count)
```

### Phase 4: Data Layer Integration

**4.1 Write ordering** (DB-first, cache-second)
```python
# CORRECT: DB is source of truth
await db.upsert_order_batch(orders)                    # 1. DB write
app_state.open_orders = await db.query_open_orders(...)  # 2. Cache from DB
order_manager.enrich_positions_tpsl(app_state.positions) # 3. Enrich cache

# WRONG: cache-first risks divergence
app_state.open_orders = orders       # ← cache updated
await db.upsert_order_batch(orders)  # ← if this fails, cache is wrong
```

**4.2 Transaction boundaries**
- Order snapshot: single transaction for all upserts + cancel marking
- Fill processing: single transaction for fill upsert + parent order update
- Position close: single transaction for closed_positions insert
- Reads: no transaction needed (SQLite WAL mode gives consistent reads)

**4.3 Cache refresh strategy**
```
Event                    → Cache Action
─────────────────────────────────────────────────
order_snapshot arrives   → rebuild open_orders from DB
                         → recompute TP/SL on positions
fill arrives             → recompute individual_fees from DB (SUM query)
                         → NO accumulation in cache
position_snapshot        → rebuild positions (preserve TP/SL from orders cache)
engine restart           → order_snapshot + position_snapshot rebuild everything
plugin disconnect        → cache stays (stale), REST fallback kicks in
```

**4.4 Idempotency guarantees**
- Orders: UNIQUE(account_id, exchange_order_id) — upsert is idempotent
- Fills: UNIQUE(account_id, exchange_fill_id) — upsert is idempotent
- Closed positions: UNIQUE(account_id, terminal_position_id, exit_time_ms) — prevents duplicate close rows
- Fees: computed via SUM query, not accumulated — inherently idempotent

**4.5 Index strategy for query patterns**
```sql
-- Open Orders tab: filter by status, sort by created_at
CREATE INDEX idx_orders_open ON orders (account_id, status, created_at_ms DESC)
    WHERE status IN ('new', 'partially_filled');

-- Order History tab: paginate by updated_at
CREATE INDEX idx_orders_history ON orders (account_id, updated_at_ms DESC);

-- Fills tab: paginate by timestamp
CREATE INDEX idx_fills_history ON fills (account_id, timestamp_ms DESC);

-- Position close: find fills for a position
CREATE INDEX idx_fills_position ON fills (terminal_position_id, is_close);

-- TP/SL lookup: find active TP/SL orders for positions
CREATE INDEX idx_orders_tpsl ON orders (account_id, symbol, position_side, order_type)
    WHERE status IN ('new', 'partially_filled');
```

| Gap | Initial Plan | Maximized Plan |
|-----|-------------|----------------|
| State machine | Implicit status strings | Explicit enum + transition validation |
| Race: fill before order | Not handled | Best-effort update, reconcile on next snapshot |
| Race: rapid snapshots | Not handled | Serial processing under lock, sequence numbers |
| Race: position close timing | Immediate | Deferred 2s to allow pending fills |
| Cache divergence | Cache-first risk | DB-first always, cache is projection |
| Fee accumulation drift | `+= fee` in cache | `SUM(fee)` from DB on every fill |
| Stale orders | Not handled | `last_seen_ms` + background staleness loop |
| Duplicate fills | Assumed unique | UNIQUE constraint + upsert dedup |
| Order amendments | Not handled | `amendment_count` + snapshot diffing |
| Invalid transitions | Not validated | State machine rejects illegal transitions |
| Partial close | Not specified | Only close on full position disappearance |
| Position flip | Not handled | Natural from correct fill tagging |
| Plugin disconnect | Not handled | Staleness detection + REST fallback |
| Testability | Not addressed | 5 test categories with specific scenarios |
| Decoupling | Logic in platform_bridge | OrderManager extracts domain logic |
