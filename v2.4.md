## Priority 1: Gate Promotion Prerequisites

dd_state is being promoted from advisory to enforcement. The audit
identified that the current intraday-only metric is insufficient for
enforcement — a trader losing 5%/day for 10 days would never trigger
"limit" if each day is below threshold.

### 1a. Monthly Drawdown Metric (MN-2-B)

**Severity**: MEDIUM | **Effort**: ~30-50 LOC | **Audit ref**: MN-2

**Problem**: Engine computes intraday drawdown only (resets at BOD).
No month-to-date drawdown exists. Dashboard label clarified to "Today's
DD" by audit (FE-MN-2), but the underlying metric gap remains.

**Fix**:
- Track `month_peak_equity` separately from daily `max_total_equity`
- Reset on 1st of month only (in `_bod_scheduler`, not daily BOD)
- Compute `monthly_drawdown = (month_peak - current) / month_peak`
- Display as "Month DD" alongside existing "Today's DD" gauge
- dd_state uses monthly metric for threshold checks

**Files**: `core/data_cache.py`, `core/state.py`, `core/schedulers.py`,
`templates/fragments/dashboard_body.html`

### 1b. Rolling-Window Drawdown for Enforcement (MN-2-C)

**Severity**: HIGH | **Effort**: ~80-120 LOC | **Audit ref**: MN-2

**Problem**: Fixed monthly window has boundary effects (loss on 30th
resets on 1st). Rolling 30-day window better reflects actual risk exposure.

**Fix**:
- Configurable rolling window (default 30 days)
- Query `account_snapshots` for peak equity in window
- dd_state uses rolling-window metric for gate enforcement
- Monthly metric (1a) as display complement, rolling as enforcement

**Depends on**: 1a (monthly metric establishes the pattern)

### 1c. Hard Circuit Breaker on Calculator

**Status**: PARTIALLY DONE (SC-2 from audit)

SC-2 already implemented `ReadyStateEvaluator` with 3 gates (bootstrap,
account, staleness) + calculator returns ineligible when not ready.
Remaining work:

- Wire dd_state "limit" into the ready-state evaluator
- Wire weekly_pnl_state "limit" into the ready-state evaluator
- Show which limit was hit and current value vs threshold
- Show warning (not block) when state is "warning"

**Files**: `core/ready_state.py`, `api/routes_calculator.py`
**Effort**: ~20-30 LOC (wiring, not new architecture)
**Depends on**: 1a/1b (metrics must be correct before gating on them)

## Priority 2: Execution Quality

### 2a. Estimated vs Actual Slippage Tracking

**Severity**: MEDIUM | **Effort**: ~100-150 LOC

**Problem**: Calculator estimates VWAP slippage from orderbook snapshot.
The estimate is never compared to actual fill price. Over 100+ trades,
systematic bias is invisible.

**Fix**: When a fill arrives, match to most recent `pre_trade_log` entry
and compute delta.

**Flow**:
```
1. Fill event arrives (symbol, fill_price, qty)
2. Query pre_trade_log: most recent calc for symbol where qty ≈ fill qty
3. Compute: actual_slippage = (fill_price - est_fill_price) / est_fill_price
4. Store in execution_log.slippage_actual (new column)
5. Over time: avg(actual - estimated) = estimator bias
```

**Note**: Audit's implementation shortfall tracking (order_manager.py
`_compute_shortfall`) already compares intended vs actual entry/exit.
This item extends that to slippage specifically.

**Files**: `core/order_manager.py`, `core/db_trades.py`,
`api/routes_analytics.py`

### 2b. Exchange Contract Spec Validation

**Severity**: MEDIUM | **Effort**: ~40-60 LOC

**Problem**: Calculator can output sizes violating exchange minimums
(min notional, min qty, lot step). Trader discovers on order rejection.

**Fix**: Validate calculated size against `exchange_info` contract specs.

```python
final_size = round_to_step(raw_size, lot_step)
if final_size * entry_price < min_notional:
    result["eligible"] = False
    result["reject_reason"] = f"Below min notional ({min_notional} USDT)"
```

**Files**: `core/risk_engine.py`, `core/exchange.py`

### 2c. Stale-Data Guard on Calculator

**Status**: DONE (SC-2 from audit)

SC-2's `ReadyStateEvaluator` checks data freshness:
- Bootstrap gate: blocks during initialization
- Staleness gate: blocks when data age > threshold (60s hysteresis)
- Calculator returns ineligible with reason when not ready

No further work needed. Verified operationally 2026-05-12.

### 2d. Position-Exchange Reconciliation on Startup

**Status**: MOSTLY DONE (MP-1 + OM-5b from audit)

- MP-1: dd_state + weekly_pnl_state restored from DB on crash recovery
- OM-5b: one-shot basic order sync on startup regardless of plugin state
- `_startup_fetch` already runs fetch_account → fetch_positions → order
  sync before `is_initializing` flips to False

Remaining: equity delta logging (if live equity differs from DB snapshot
by >1%, log WARNING). ~5 LOC.

## Priority 3: Rate-Limit Architecture

### 3a. Fan-Out Architectural Review (RL-4-B)

**Severity**: MEDIUM | **Effort**: ~60-100 LOC | **Audit ref**: RL-4

**Problem**: 6 independent subsystems respond to one trade-close event,
each making separate REST calls. RL-4 serialized them with Semaphore(2),
but the underlying fan-out is architectural duplication.

**Fix**: Single coordinated REST sweep on trade close, distribute results
to subscribers via event payload. Reduces fan-out from 6 independent REST
calls to 1 coordinated sweep.

**Files**: `core/reconciler.py`, `core/ws_manager.py`, `core/exchange.py`

### 3b. Proactive Weight Tracker (RL-2)

**Severity**: MEDIUM | **Effort**: ~100-200 LOC | **Audit ref**: RL-1

**Problem**: Engine reacts to 429 errors (RL-1 pause mechanism) but
doesn't prevent them. Proactive tracking would count API weight and
throttle BEFORE hitting the limit.

**Fix**: Track cumulative weight per minute, pause proactively when
approaching budget ceiling.

**Files**: `core/exchange.py`, adapter wrappers

## Priority 4: UI Architecture

### 4a. HTMX Position Card Refresh (FE-3)

**Severity**: MEDIUM | **Effort**: ~30-50 LOC | **Audit ref**: FE-3

**Problem**: Dashboard refreshes entire `#dashboard-body` every 1s via
`hx-swap="innerHTML"`, causing full re-render of positions, orders, and
history panels.

**Options**:
- HTMX morphing extension (`hx-swap="morph:innerHTML"`) — DOM diffing
  instead of full replacement
- Split into fragment endpoints — positions, orders, history each refresh
  independently
- Both approaches depend on v2.4 UI architecture decision

### 4b. Ghost Interface Pattern (UX-1)

**Severity**: MEDIUM | **Audit ref**: UX-1

Three states for data-dependent components:
- Engine unreachable — ghost structure with "data pending" indicators
- Data stale (SC-2 not-ready) — ghost overlay/fade with staleness annotation
- Data immature / bootstrapping — consistent loading indicators

Cross-cuts FE-2, FE-3, SC-2. Consolidated design after FE-3 resolves.

### 4c. Unified Data Table Pattern (UX-2)

**Severity**: MEDIUM | **Audit ref**: UX-2

Pagination + sort + filter + search as reusable component, applied
selectively:
- HIGH benefit: Traded Pairs analytics, MFE/MAE per-trade view
- MEDIUM benefit: Open Orders (sort by age/symbol, filter basic vs conditional)
- LOW benefit: Open Positions (typically few rows, full pattern overkill)

Depends on v2.4 UI architecture decision (HTMX continuation vs framework
upgrade). Redis pub/sub + WebSocket push may favor component framework.

### 4d. Engine Responsiveness (FE-11)

**Severity**: MEDIUM | **Audit ref**: FE-11

Rapid timeframe switching makes engine unresponsive (synchronous backend
blocking event loop during query/recompute). Properly addressed by Redis
event-driven state architecture. Band-aid (debounce) possible but hides
the real problem.

### 4e. TP/SL Columns on Order Rows (FE-14)

**Severity**: LOW | **Effort**: ~20-30 LOC | **Audit ref**: FE-14

Display tpTriggerPrice/slTriggerPrice for entry orders with attached
TP/SL legs. Needs DB schema addition (`tp_trigger_price`, `sl_trigger_price`
columns on orders table) + adapter parsing change.

## Priority 5: Data Quality

### 5a. One-Way Mode Break-Even Close Fix (AD-4-B)

**Severity**: LOW | **Audit ref**: AD-4

AD-4 fixed is_close for hedge mode (deterministic side+positionSide).
One-way mode falls back to realizedPnl heuristic — break-even closes
(PnL=0) are misclassified. Fix requires position-direction context at
trade time (lookup by symbol → direction at that moment).

### 5b. Equity History Backfill (MP-2)

**Severity**: MEDIUM | **Audit ref**: MP-2

Engine equity curve shows flat line during downtime. No backfill mechanism
for equity gaps. Options: compute from price+state during downtime, derive
from exchange equity endpoint, or mark gaps explicitly in UI.
