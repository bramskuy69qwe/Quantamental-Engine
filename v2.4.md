## Priority 0: Decisions Made for v2.4

Decisions baked in before implementation starts. These resolve open
questions surfaced during planning review and govern how subsequent
priorities are scoped. Claude Code sessions implementing v2.4 should
treat this section as authoritative spec.

### 0a. Timezone wiring

- **Source of truth**: new `account_settings.timezone` column (default `UTC`).
- **Coverage**: every calendar-driven boundary (DD reset, weekly P&L window,
  any other reset) reads TZ from `account_settings`. No hardcoded TZ in
  engine logic.
- **Change semantics**: TZ changes apply **prospectively** only. The next
  reset boundary uses the new TZ; the current period continues under the
  old TZ. No retroactive state rewrites.
- **Backtest semantics**: TZ is an explicit parameter in
  `backtest_runs.params_json`, defaulted to current account TZ on the
  dispatcher form but persisted per run. Required for reproducibility —
  changing account TZ later must not silently change old backtest outputs.
- **Hardcoded TZ audit**: as part of 1a, grep `utcnow`, `astimezone`,
  `datetime.now(`, `pytz`, `zoneinfo` across `core/`. Each site labeled
  in a comment as either "intentional UTC for exchange interop" or
  "account-TZ-bound, reads from settings".

### 0b. UI architecture decision: Redis pub/sub + HTMX morphing

- Redis pub/sub replaces in-process `core/event_bus.py` as the canonical
  event bus.
- HTMX morphing (idiomorph extension) replaces `hx-swap="innerHTML"` for
  fragment refreshes.
- These are **complementary, not alternatives**. Redis decouples event
  production from consumption (also serves v2.5+ model registry). HTMX
  morphing optimizes DOM-side rendering inside the existing template
  architecture.
- **No component framework upgrade** (React/Vue/Svelte). Server-rendered
  HTMX stays. Existing 25+ Jinja fragments preserved.
- **Migration**: 2-4 week dual-publish window. Every event publishes to
  both buses. UI subscribers migrate one at a time. After all subscribers
  on Redis, in-process bus removed.

### 0c. Enforcement rollout pattern

All advisory→enforcement transitions ship in **shadow mode** first.

- New config flag `account_settings.dd_enforcement_mode: advisory|enforced`,
  default `advisory`.
- Shadow period: minimum 2-3 weeks of computing the new metric and logging
  would-have-blocked events without actually blocking.
- Review surface: `/admin/shadow_events` route lists every would-have-blocked
  event in window.
- Flip to `enforced` is a UI action with confirmation; writes an
  `engine_events` row.
- **Rollback**: flip the flag back to `advisory`. No code revert needed.

### 0d. Adapter capability flags

- Adapter protocol gains a `capabilities: dict[str, bool]` attribute.
- Initial keys: `orders`, `conditional_orders`, `market_data`,
  `account_query`, `position_query`, `historical_equity`.
- Engine factory checks capabilities at wire time. Calculator refuses to
  instantiate for accounts whose adapter lacks `"orders"`.
- v2.4.5 MEXC adapter is the first user of the pattern
  (read-only = `orders: False`).

### 0e. Config storage policy

- **Per-account knobs** (TZ, DD thresholds, DD rolling window, enforcement
  mode, weight budget %, equity-recovery threshold, analytics default
  period, strategy preset) → `account_settings` table.
- **Infrastructure config** (Redis URL, encryption keys, DB paths) →
  env vars / `.env`.
- **No hot-reload.** Restart on config change. Acceptable for a
  single-machine engine; removes the "did the setting take effect?"
  debugging class.

### 0f. Migration discipline

- Every schema change ships as a numbered file under `core/migrations/`.
- v2.4 migrations follow the convention `0XX_v2_4_<feature>.sql`.
- Coordinated migration list in Priority 6c.
- Each migration ships with a corresponding test under `tests/migrations/`.

### 0g. Cardinality decision: 1 calc → 1 logical position

- Each calculator call produces one intended position. `calc_id` is 1:1
  with positions.
- Scale-ins are separate calculator calls with separate `calc_id`s.
- Keeps slippage analytics joins simple (calc → fills, no fan-out).

### 0h. Strategy presets

Account creation form picks a strategy preset that seeds the
risk-related `account_settings` columns. Settings remain individually
editable after creation — preset is a starting point, not a lock.

Rationale: different trading styles have different natural drawdown
accumulation rates. A scalper's bad streak compresses into days; a swing
trader's unfolds over weeks. The gate metric (rolling-window DD) is the
same — only the window length and thresholds differ.

**Preset → seeded values:**

| Preset | `dd_rolling_window_days` | `dd_warning_threshold` | `dd_limit_threshold` | `dd_recovery_threshold` | `analytics_default_period` |
|--------|--------------------------|------------------------|----------------------|-------------------------|----------------------------|
| `scalping` | 14 | 0.04 | 0.08 | 0.50 | `weekly` |
| `day_trading` | 21 | 0.05 | 0.10 | 0.50 | `weekly` |
| `swing` | 45 | 0.06 | 0.12 | 0.40 | `monthly` |
| `position` | 90 | 0.08 | 0.15 | 0.30 | `quarterly` |
| `custom` | — | — | — | — | — |

Defaults above are placeholders to be tuned from your own trading
history during shadow period (Priority 1e).

Stored as `account_settings.strategy_preset` for the record (informational
— the preset name doesn't drive logic after seeding; the individual
settings do).

### 0i. Analytics period preference

Separate from the DD gate window. Drives **display defaults** across the
UI, not risk policy.

- New column `account_settings.analytics_default_period` with values:
  `weekly`, `monthly`, `quarterly`, `yearly`, `rolling_30d`,
  `rolling_90d`, `all_time`.
- Drives default time-range / pagination on:
  - Position history pagination window
  - P&L summary default time range
  - Performance breakdown grouping (week / month / quarter / year buckets)
  - Equity curve default zoom
  - Traded Pairs analytics default window
  - (v2.5+) Backtest results default time range
- All consuming views expose a dropdown to switch on demand. The setting
  is the default, not a constraint.

Calendar period boundaries (week start day, fiscal year offset) use the
account TZ from 0a. Week start defaults to Monday; overridable via
`account_settings.week_start_dow` if needed (deferred decision —
add only if a user actually wants Sunday-start).

## Priority 1: Gate Promotion

dd_state is being promoted from advisory to enforcement.

### 1a. Rolling-Window Drawdown Metric

**Severity**: HIGH | **Effort**: ~100-150 LOC | **Audit ref**: MN-2

Replaces the v2.3 intraday-only DD metric. Calendar-month DD is **dropped
from v2.4** — rolling-window is the single metric for both display and
enforcement. (If a calendar view is needed later for reporting/tax, it
becomes a separate analytics page in v2.5+, not a competing dashboard
gauge.)

**Problem**: Engine computes intraday drawdown only (resets at BOD). A
trader losing 5%/day for 10 days never triggers "limit" because each day
stays below threshold.

**Fix**:
- Configurable rolling window (default 30 days for accounts created
  without a preset; otherwise seeded from strategy preset per 0h) in
  `account_settings.dd_rolling_window_days`.
- Query `account_snapshots` for peak equity in window.
- `rolling_drawdown = (rolling_peak - current) / rolling_peak`.
- Thresholds: `account_settings.dd_warning_threshold` (default 0.05),
  `account_settings.dd_limit_threshold` (default 0.10). Both seeded from
  strategy preset if used at account creation.
- dd_state uses rolling-window metric exclusively.

**Performance**: at 1-minute snapshot cadence, 30-day window ≈ 43K rows
per query. **Strategy**: cache `rolling_peak_equity` in `data_cache`,
invalidate on snapshot write. Simpler than maintaining a pre-aggregated
5-minute summary table. Fall back to 5-minute summary if cache logic
becomes gnarly (decide during implementation, not now).

**Files**: `core/data_cache.py`, `core/state.py`, `core/schedulers.py`,
`templates/fragments/dashboard_body.html`,
`core/migrations/0XX_v2_4_rolling_dd.sql`

### 1b. Recovery Semantics

**Severity**: HIGH | **Effort**: ~40-60 LOC

When `dd_state == "limit"`, blocking persists until one of:

- **Calendar-boundary recovery**: next rolling window reset boundary
  passes. Default behavior.
- **Equity-recovery early-unblock**: equity rebuilds to within
  `account_settings.dd_recovery_threshold` (default 0.50) of the
  drawdown peak. Example: peak = 10000, trough = 9000 (10% DD), with 0.50
  threshold unblock fires when equity ≥ 9500.
- **Manual override**: trader explicitly overrides via UI. Writes an
  `engine_events` row with required reason text. 5-second confirmation
  pause built into the UI (the pause is the feature — catches tilt).

Auto-unblock conditions evaluated on every snapshot write.

**Files**: `core/risk_engine.py`, `core/ready_state.py`,
`templates/fragments/dd_override.html`, `api/routes_account.py`

### 1c. Open-Position Behavior on Limit Hit

**Severity**: HIGH | **Effort**: ~30-50 LOC

When `dd_state == "limit"`:

- **Block**: new entry orders via calculator (1d wires this).
- **Allow**: TP/SL modifications on existing positions. Volatility
  expansion may require widening stops; blocking this would force
  losses.
- **Allow**: reduce-only orders / position closes. Exits must always
  be possible.
- **Do not**: auto-flatten existing positions. That's a separate policy
  decision (portfolio auto-stop-loss), out of scope for v2.4.

**Files**: `core/order_manager.py`, `core/ready_state.py`

### 1d. Hard Circuit Breaker on Calculator

**Status**: PARTIALLY DONE (SC-2 from audit) — wiring remains.

**Severity**: HIGH | **Effort**: ~20-30 LOC

- Wire `dd_state == "limit"` into `ReadyStateEvaluator` (blocks calculator).
- Wire `weekly_pnl_state == "limit"` into `ReadyStateEvaluator`.
- Block message shows which gate fired and current value vs threshold.
- Warning state (`warning`) shows a banner but doesn't block.

**Files**: `core/ready_state.py`, `api/routes_calculator.py`

### 1e. Shadow→Enforce Rollout Infrastructure

**Severity**: HIGH | **Effort**: ~50-80 LOC

Per Priority 0c.

- Default: v2.4 ships with `dd_enforcement_mode: advisory`.
- Engine computes new rolling metric and recovery logic regardless of
  mode.
- In `advisory` mode: would-have-blocked events write to `engine_events`
  with `event_type: would_have_blocked_dd` but the calculator returns
  eligible.
- In `enforced` mode: same logic actually blocks.
- `/admin/shadow_events` route renders the would-have-blocked log.
- Mode flip UI requires typing the account name to confirm.

**Files**: `core/ready_state.py`, `api/routes_admin.py` (new),
`templates/admin/shadow_events.html`

## Priority 2: Execution Quality

### 2a. Slippage Tracking via calc_id Propagation

**Severity**: MEDIUM | **Effort**: ~150-200 LOC

**Problem**: Calculator estimates VWAP slippage from orderbook snapshot.
Estimate is never compared to actual fill. Over 100+ trades, systematic
bias is invisible. Audit's original "match by qty ≈ fill qty" heuristic
is fragile (float comparison, race conditions, manual-order
contamination); replaced with identity propagation.

**Architecture**:

1. Calculator generates a `calc_id` (UUID) when producing a size
   recommendation. Stored on `pre_trade_log.calc_id`.
2. `calc_id` rides with the order ticket through `order_manager` and
   into the exchange's `clientOrderId` field (prefix or embedded).
3. Local orders table stores `calc_id` alongside `clientOrderId`.
4. Each fill writes one `execution_log` row with `calc_id` resolved from
   the originating order. Per-fill granularity preserved (Option B —
   aggregation reversible from per-fill data; reverse not true).
5. Analytics joins `execution_log` to `pre_trade_log` on `calc_id`. No
   heuristic matching.

**Per-fill row schema**:

- One row per partial fill.
- `slippage_actual = (fill_price - est_fill_price_at_calc_time) / est_fill_price_at_calc_time`.
- `fill_type` tagged: `entry`, `tp`, `sl`, `manual`, `reduce_only`.
  Entry vs TP/SL slippage have different distributions and must not be
  averaged together.
- Manual orders (no `calc_id`) get `calc_id = NULL`, filtered from
  slippage analytics.

**Position-level slippage**: computed at query time as VWAP-weighted
aggregate of constituent fills. Not pre-aggregated.

**Downstream**: v2.6 offensive backtest's slippage model reads per-fill
data conditioned on fill size. Heuristic matching would produce a noisy
biased distribution; calc_id propagation is the architectural minimum
for honest backtest slippage modeling.

**Files**: `core/risk_engine.py` (generate calc_id), `core/order_manager.py`
(propagate), `core/db_trades.py` (storage), `api/routes_analytics.py`
(aggregate views), `core/migrations/0XX_v2_4_calc_id.sql`,
`core/migrations/0XX_v2_4_slippage_actual.sql`

### 2b. Exchange Contract Spec Validation

**Severity**: MEDIUM | **Effort**: ~60-100 LOC

**Problem**: Calculator can output sizes violating exchange minimums
(min notional, min qty, lot step). Trader discovers on order rejection.

**Fix**:
- All lot-step rounding uses `decimal.Decimal`, not float. Float
  arithmetic on lot step has cost production trading systems real
  money — `0.1 + 0.2 != 0.3`.
- `exchange_info` cached per-adapter with explicit TTL (default 1 hour).
  Refresh triggers: TTL expiry OR adapter receives a contract-related
  error from exchange.
- Validation order: round to lot_step → check min_qty → check
  min_notional → check max_qty.
- On rejection, calculator returns ineligible with `reject_reason` AND
  `suggested_size` (nearest valid size that satisfies all constraints).
  UI shows both.

**Files**: `core/risk_engine.py`, `core/exchange.py`,
`core/adapters/binance/`, `core/adapters/bybit/`

### 2c. Stale-Data Guard on Calculator

**Status**: DONE (SC-2 from audit). No further work.

### 2d. Position-Exchange Reconciliation on Startup

**Status**: MOSTLY DONE (MP-1 + OM-5b from audit).

Remaining: equity delta logging. If live equity differs from DB snapshot
by >1%, log WARNING and write `engine_events` row of type
`equity_delta_warning`. ~10 LOC.

## Priority 3: Rate-Limit Architecture

### 3a. Fan-Out Coordination Review

**Severity**: MEDIUM | **Effort**: ~80-120 LOC | **Audit ref**: RL-4

**Problem**: Multiple subsystems independently respond to one trade-close
event with separate REST calls. RL-4 serialized them with Semaphore(2)
but the architectural duplication remains.

**Decision**: single coordinated REST sweep on trade close, results
distributed via event payload.

**Required design step (Phase 4 entry)**: enumerate the subscribers
driving fan-out. Investigation deliverable before writing the coordinated
sweep:

- `reconciler.py` — what does it fetch on trade close?
- `data_cache.py` — what does it refresh?
- `order_state.py` — TP/SL re-evaluation triggers?
- `state.py` — equity recompute?
- Analytics writers?
- UI push triggers (Redis-driven post-Priority 4)?

Once enumerated, the coordinated sweep design includes
**per-subscriber timeout + isolation** — a slow subscriber doesn't block
fast ones; on timeout the slow subscriber is logged and dropped from
that sweep cycle.

**Files**: `core/reconciler.py`, `core/ws_manager.py`, `core/exchange.py`

### 3b. Proactive Weight Tracker

**Severity**: MEDIUM | **Effort**: ~150-200 LOC

**Architecture**:
- Tracker lives **per-adapter** — each exchange has its own rate-limit
  model (Binance is weight + IP-bound + headers; Bybit is
  requests-per-second per-endpoint + headers; MEXC differs again).
- Implementation: tracker is part of the adapter singleton from
  `exchange_factory`. **Shared across all API keys hitting that adapter
  from this engine instance** (one IP from Warp = one weight budget
  regardless of how many account keys route through it).
- **Server-side weight** (response headers, e.g. `X-MBX-USED-WEIGHT-1M`)
  is source of truth.
- **Client-side counting** is the running estimate. Reconcile on every
  response: server header overrides estimate.
- Configurable budget ceiling per adapter
  (`adapter_settings.weight_budget_pct`, default 0.80). Pause REST
  scheduling when usage hits ceiling.
- Disable via `adapter_settings.weight_tracker_enabled: false` for
  rollback.

**Endpoint weight registry** (new): `core/adapters/<exchange>/weights.py`
maps endpoint → weight. Maintained alongside the adapter doc.

**Files**: `core/adapters/binance/`, `core/adapters/bybit/`,
`core/adapters/protocols.py` (add weight tracking surface), adapter docs

### 4a. Redis Pub/Sub Migration

**Severity**: HIGH | **Effort**: ~300-500 LOC + ops setup

`feature_flags.event_bus_backend: in_process|redis` flag flip.

## Priority 5: Data Quality

### 5a. One-Way Mode Break-Even Close Fix

**Severity**: LOW | **Effort**: ~80-120 LOC (larger than originally scoped)
| **Audit ref**: AD-4

AD-4 fixed `is_close` for hedge mode (deterministic side + positionSide).
One-way mode falls back to a `realizedPnl` heuristic — break-even closes
(PnL = 0) are misclassified.

**Note on scope**: original v2.4 plan described this as a simple lookup.
Correct fix requires position-direction context at trade time, which
needs either:

- (a) Position-snapshot table indexed by `(symbol, timestamp)` to join
  against during trade ingestion. (go-forward)
- (b) Backfill pass: walk trade ledger from beginning, reconstruct
  position direction sequentially. (one-shot historical correction)

Both needed because v2.6 strategy-regime heatmap consumes historical
`is_close` labels — biased history produces a biased heatmap. **Move
this priority forward** if v2.5 start date is close to v2.4 ship.

**Files**: `core/order_state.py`, `core/db_trades.py`,
`scripts/backfill_isclose.py`,
`core/migrations/0XX_v2_4_position_snapshots.sql`

### 5b. Equity History Backfill

## Priority 6: Observability & Migration Coordination

Cross-cutting infrastructure used by every other priority.

### 6a. engine_events Table

**Severity**: HIGH (load-bearing for risk policy audit) | **Effort**: ~80-120 LOC

Single audit-trail table for all stateful behavior the trader needs to
retrospect.

**Schema**:
```sql
CREATE TABLE engine_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  account_id INTEGER NOT NULL,
  event_type TEXT NOT NULL,
  payload_json TEXT NOT NULL,
  timestamp DATETIME NOT NULL,
  source TEXT NOT NULL
);
CREATE INDEX idx_engine_events_account_time
  ON engine_events(account_id, timestamp DESC);
CREATE INDEX idx_engine_events_type ON engine_events(event_type);
```

**Initial event types**:
- `dd_state_transition` (warning ↔ limit ↔ ok)
- `calculator_blocked` (block reason, gate that fired)
- `manual_override` (gate overridden, reason text)
- `enforcement_mode_change` (advisory ↔ enforced)
- `equity_delta_warning` (live vs DB snapshot mismatch >1%)
- `rate_limit_pause` (proactive weight tracker engaged)
- `would_have_blocked_dd` (shadow mode event)
- `would_have_blocked_weekly_pnl` (shadow mode event)

**Surface**: `/admin/events` route, fragment with last 50 events, filter
by type and date range.

**Files**: `core/event_log.py` (new), `api/routes_admin.py`,
`templates/admin/events.html`,
`core/migrations/0XX_v2_4_engine_events.sql`

### 6b. State-Machine Test Category

**Severity**: HIGH | **Effort**: ~150-250 LOC (10-15 fixtures + harness)

New `tests/state_machine/` directory for behavioral coverage of dd_state.
Smoke-diff covers math regressions; state-machine tests cover behavior
regressions in stateful logic.

**Fixture format**: CSV of `(timestamp, equity)` pairs + expected state
at checkpoint timestamps + override events at specified timestamps.

**Initial fixtures**:
- Slow drift to limit (gradual DD over 20 days)
- Sharp drop within window (single bad day triggers limit)
- Recovery with early-unblock (DD then partial recovery to threshold)
- Recovery without early-unblock (DD then sideways, hits boundary)
- Multi-cycle (limit → recover → limit again)
- TZ boundary case (reset happens during a drawdown — verify peak
  preserved across the boundary)
- Override path (manual override fires; subsequent events logged)
- Shadow mode (would-have-blocked logged, no actual block)

Harness uses synthetic clock from `core/test_clock.py` (new) to advance
engine time deterministically.

**Files**: `tests/state_machine/`, `core/test_clock.py`

## Risk Register

| Priority | Risk | Rollback |
|---------|------|----------|
| 1 (Gate promotion) | False-positive blocks during live trading | Flip `dd_enforcement_mode` to `advisory` in `account_settings`. No code revert. Shadow-mode period catches most cases pre-release. |
| 2a (calc_id propagation) | Bug in propagation breaks order placement | Migration adds nullable columns; old code paths unaffected. Validate calc_id propagation in advisory mode before relying on it for analytics. Rollback = stop populating calc_id on new orders; existing rows unaffected. |
| 2b (contract validation) | Stale `exchange_info` cache rejects valid sizes | Force-refresh trigger on contract error; TTL configurable. Rollback = revert to pre-validation (calculator returns size, exchange rejects on bad input). |
| 3 (rate limit) | Proactive tracker pauses too aggressively, blocks legitimate REST traffic | Configurable budget ceiling (default 0.80). Lower to 0.60 if false-pauses appear; raise toward 0.90 if too conservative. Disable via `adapter_settings.weight_tracker_enabled: false`. |
| 4a (Redis) | Event delivery breaks during migration | Dual-publish window keeps in-process bus running. Subscribers revertible individually. After dual-publish closes, `feature_flags.event_bus_backend: in_process` re-enables old path. |
| 4b / 4c (HTMX) | Morphing extension breaks form state / focus / animations | Revert `hx-swap` attributes; fall back to `innerHTML`. ~15 min revert. |
| 5a (is_close backfill) | Backfill script produces wrong direction labels at scale | Run on a copy of the DB first. Compare labeled output to a manually-verified sample. Only apply to prod DB after sample validates. |
| 6a (engine_events) | Write contention from high-frequency event types | Indexes on `(account_id, timestamp DESC)` and `event_type`. Async writes via dedicated logger. If contention surfaces, batch writes via queue. |
| v2.4.5 (MEXC) | Adapter exposes engine-core leak during integration | Capability flags should prevent this. If a leak surfaces, halt MEXC integration, fix the abstraction first. |
