# v2.5–v2.7 — Backtesting Subsystem, Defensive ML & Integration Backtest

## Context

This roadmap extends the v2.4 planning artifact with three forward
versions building out the backtesting and model management subsystem.
The v2.4 release is preserved as-is; this document specifies v2.5–v2.7
scope and the cross-version dependencies that link v2.4 items to
downstream backtest work.

The backtesting subsystem follows the established **core + adapter
ring** pattern. Engine-core defines vendor-neutral backtest protocols;
specific tools (native, VectorBT Pro, optionally NautilusTrader)
implement them. No engine-core code is duplicated between live and
historical replay — `core/regime_classifier.py` is reused as-is across
both paths.

The defensive regime classifier progresses from rule-based (current,
Stage A) through parametric statistical (HMM, Stage B), supervised ML
(Stage C, optional), to hybrid (rule-based definitions + ML estimation
+ rule-based actions, Stage D). Stage A is registered as v1 in v2.5;
Stage B is v2 in v2.6; Stage D is v3 in v2.7. Stage C is evaluated
empirically and only built if Stage D needs additional capacity.

**Source documents**: `v2.4.md`, `docs/audit/AUDIT_CLOSEOUT.md`, `README.md`

## Architectural addendum

### Adapter ring extension

A new `core/adapters/backtest/` ring is added alongside the existing
exchange and platform adapters. Three protocols are defined:

- `OffensiveBacktestAdapter` — strategy simulation (entries, exits, P&L)
- `DefensiveBacktestAdapter` — regime classifier validation, risk system simulation
- `IntegrationBacktestAdapter` — full-engine replay (offensive + defensive + risk caps + DD enforcement all running together against historical data)

Each protocol exposes `run(params, data_window, model) -> BacktestRunHandle`,
`get_metrics(run_id) -> dict`, and `get_trades(run_id) -> list[BacktestTrade]`.

### Model registry

`core/models/registry.py` provides versioned model management. Every
model (offensive or defensive) is registered with a unique `model_id`,
incrementing `version`, and stored artifact at
`data/models/{model_id}/{version}/`:

```
manifest.json         # model_type, version, params, training_window,
                      # training_data_hash, performance_summary
model.pkl             # serialized model object
notes.md              # free-form
```

The existing rule-based `core/regime_classifier.py` is wrapped as
`defensive.regime_classifier` v1 in v2.5. New versions register
alongside without replacing v1, enabling A/B comparison.

### Calculator integration

In v2.6, the calculator records `model_id` and `model_version` per
pre-trade calculation. Live metrics (Sharpe, win rate, expectancy) are
computed nightly from `execution_log` joined to model versions and
written to `model_live_metrics`. The calculator UI surfaces backtest
metrics next to live metrics per model, enabling drift detection.

## v2.4 — Recap and forward dependencies

The v2.4 release is unchanged. Three items have explicit v2.5+ implications.

### v2.4 → v2.5+ dependency: 2a Slippage tracking (v2.4 Priority 2)

**Forward use**: the empirical slippage distribution feeds the v2.6
offensive backtest as the slippage model. Without 2a, backtest slippage
assumptions reduce to flat constants and the backtest cannot honestly
estimate per-trade slippage at the symbol/size granularity that matters.

### v2.4 → v2.5+ dependency: 1a/1b Monthly + rolling DD metrics (v2.4 Priority 1)

**Forward use**: the metrics validated by the v2.5 defensive backtest.
"Would the dd_state limit have fired in this historical period?" is
unanswerable without correct DD calculation. v2.5's defensive backtest
runs validate the metric implementations, not the other way around.

### v2.4 → v2.5+ dependency: 4d Engine responsiveness via Redis (v2.4 Priority 4)

**Forward use**: Redis pub/sub infrastructure used by the v2.5 model
registry to publish regime updates and model state changes. v2.5 can
fall back to file-based notification if 4d slips, but at higher
implementation complexity and added I/O latency.

**No v2.4 items are blocked by v2.5–v2.7.** The backtest subsystem is
strictly downstream of the v2.4 release.

## v2.5 — Backtesting Foundation

Foundation work: protocols, schemas, model registry, and the first
backtest adapter (defensive native). No engine-core changes. Frontend
adds three new pages (model registry, defensive backtest dispatcher,
results view).

### Priority 1: Protocols & schema design

#### 1a. Backtest adapter protocols

**Severity**: HIGH | **Effort**: ~50-100 LOC (protocol only)

Three Python protocols defined in `core/adapters/backtest/protocols.py`:

```python
class OffensiveBacktestAdapter(Protocol):
    async def run(
        self, params: dict,
        data_window: tuple[datetime, datetime],
        model: OffensiveModel,
    ) -> BacktestRunHandle: ...
    async def get_metrics(self, run_id: str) -> dict: ...
    async def get_trades(self, run_id: str) -> list[BacktestTrade]: ...

class DefensiveBacktestAdapter(Protocol): ...
class IntegrationBacktestAdapter(Protocol): ...
```

Tag VERIFIED on the data contract, ASSUMED on adapter behavior (no
implementations yet).

**Files**: `core/adapters/backtest/protocols.py`,
`docs/adapters/backtest_protocols.md`

#### 1b. Model protocols

**Severity**: HIGH | **Effort**: ~30-50 LOC

```python
class Model(Protocol):
    model_id: str
    version: int
    params: dict
    metadata: dict
    async def predict(self, features: FeatureVector) -> Prediction: ...

class DefensiveModel(Model, Protocol):
    async def predict(self, features: FeatureVector) -> RegimeLabel: ...

class OffensiveModel(Model, Protocol):
    async def predict(self, features: FeatureVector) -> Signal: ...
```

**Files**: `core/models/protocols.py`

#### 1c. Database schema migration

**Severity**: HIGH | **Effort**: ~150-200 LOC (migration + verification)

New tables added via migration following existing discipline in
`core/migrations/`:

- `model_registry` — model_id, model_type, version, params_json, artifact_path, created_at, deprecated_at
- `backtest_runs` — run_id, adapter_type, model_id, model_version, params_json, data_version_hash, code_commit, random_seed, started_at, finished_at, status
- `backtest_metrics` — run_id, metric_name, value, computed_at
- `backtest_trades` — run_id, timestamp, symbol, side, entry_price, exit_price, qty, pnl, regime_label_at_entry
- `model_live_metrics` — model_id, model_version, window, metric_name, value, computed_at
- `data_versions` — version_hash, source_table, row_count, computed_at

Smoke-diff baseline updated to include the new tables (empty diff on
clean migration).

**Files**: `core/migrations/0XX_backtest_tables.sql`,
`tests/migrations/test_0XX_backtest_tables.py`

### Priority 3: Defensive backtest (native adapter)

#### 3a. Native defensive backtest adapter

**Severity**: HIGH | **Effort**: ~400-600 LOC

`core/adapters/backtest/native_defensive/` implements
`DefensiveBacktestAdapter`. Replays a defensive model against historical
features and produces three output layers:

1. **Classifier quality** — out-of-sample accuracy (vs forward-return-derived ground truth), stability (regime persistence), average duration, transition matrix.
2. **Market-conditional metrics** — within-regime forward-return distributions (mean, std, skew of next 1d / 5d / 20d BTC returns per regime).
3. **Strategy-regime heatmap (skeleton)** — joins `execution_log` to regime labels, computes Sharpe / win rate / expectancy per (model, regime). Skeleton in v2.5 because offensive models aren't registered yet; populated fully in v2.6.

Walk-forward implementation: configurable train / test windows, default
24m train / 3m test, rolling by 1m.

**Files**: `core/adapters/backtest/native_defensive/`,
`docs/adapters/backtest_defensive_native.md`

#### 3b. Backtest orchestration

**Severity**: MEDIUM | **Effort**: ~150-200 LOC

`core/backtest/orchestrator.py` coordinates backtest runs: validates
params, allocates run_id, dispatches to adapter, persists results,
handles failures and partial completions.

**Files**: `core/backtest/orchestrator.py`, `core/backtest/results.py`

### Priority 4: Reproducibility & audit

#### 4a. Backtest run record

**Severity**: HIGH | **Effort**: ~80-120 LOC

Every backtest run records to `backtest_runs`:
- `data_version_hash` — sha256 of (input_data_paths, last_modified_timestamps)
- `code_commit` — git rev-parse HEAD at run start
- `random_seed` — explicit, never None
- `params_json` — full parameter set, including defaults

Re-running with the same record reproduces identical metrics. Verified
in test: `assert backtest_run(params).metrics == backtest_run(params).metrics`.

**Files**: `core/backtest/reproducibility.py`,
`tests/backtest/test_reproducibility.py`

#### 4b. Smoke-diff baseline for backtests

**Severity**: MEDIUM | **Effort**: ~100-150 LOC

Extends the existing 111-row smoke-diff suite with a deterministic
defensive backtest baseline. Fixed input data, fixed params, fixed seed;
output committed as `tests/smoke_baseline/defensive_backtest_baseline.json`.
Any change in backtest math produces a non-empty diff.

**Files**: `tests/smoke_baseline/defensive_backtest_baseline.json`,
`scripts/smoke_diff.py`

### Priority 5: Frontend

**Severity**: MEDIUM | **Effort**: ~150-200 LOC HTML + routes

## v2.6 — Defensive ML, Offensive Backtest, Calculator Integration

Defensive classifier progresses to Stage B (HMM / GMM). First offensive
backtest adapter (VectorBT Pro) implemented. Calculator surface
integrates with model registry; per-model live metrics computed nightly.
Strategy-regime heatmap populated with real offensive results.

### Priority 1: Defensive ML model (Stage B)

#### 1a. HMM defensive model implementation

**Severity**: HIGH | **Effort**: ~300-500 LOC

`core/models/defensive/hmm_v1/` implements `DefensiveModel` protocol
using `hmmlearn`. 5 hidden states matching the existing rule-based
regime taxonomy (risk-on trending, risk-on ranging, risk-off ranging,
risk-off trending, vol expansion).

Training: walk-forward expanding window, 24m initial train, 3m test,
1m roll. State labels assigned post-hoc by matching state-conditional
forward-return characteristics to the named regimes.

Registered as `defensive.hmm` v1.

**Files**: `core/models/defensive/hmm_v1/`, `core/models/training/`,
`docs/models/defensive_hmm.md`

## v2.7 — Hybrid Defensive, Integration Backtest, Maturation

Hybrid Stage D model. Integration backtest replays the full engine
against historical data. Walk-forward retrain scheduler automates
defensive model versioning. Drift detection runs continuously.

**Scope warning**: this version is overscoped as drafted. Realistic
split: v2.7 covers Priorities 1–3 (hybrid + integration + walk-forward);
v2.8 covers Priorities 4–6 (drift detection + data versioning +
multi-model). Split decision deferred to v2.6 closeout.

### Priority 1: Hybrid defensive model (Stage D)

Rule-based regime definitions (from rule_based_v1), ML-based regime
probability estimation (from hmm_v1 or supervised model if added),
rule-based actions per regime (current engine architecture). Registered
as `defensive.hybrid` v1.

The hybrid model's interpretability advantage: every decision
traceable to "rule X says we're in regime A; ML probability is 78%;
action Y applies." Compare to pure ML which produces predictions
without explanation.

**Effort**: ~300-500 LOC.

### Priority 2: Integration backtest adapter

Native implementation: replays the entire engine against historical
data with `time` mocked and `ExchangeAdapter` swapped for a
`HistoricalExchangeAdapter`. Outputs full reconstruction of what the
engine would have done given current rules.

Used for regression-testing rule changes ("change dd_state threshold,
replay last quarter, did any prior session get wrongly blocked?") and
for multi-model interaction analysis ("would Models A, B, C have
competed for capital under the current portfolio cap?").

**Effort**: ~600-1000 LOC.

### Priority 3: Walk-forward retrain scheduler

Background scheduler refits defensive models on cadence (default
quarterly) or on regime-shift detection. Promotes new version through
shadow → A/B → live workflow:
- **Shadow**: new version computes predictions but doesn't drive actions
- **A/B**: new version drives actions for X% of cases (configurable)
- **Live**: full promotion, old version deprecated

**Effort**: ~300-400 LOC.

### Priority 4: Drift detection

Nightly scheduler compares live metrics to backtest metrics per
registered model. Surfaces alerts on dashboard when live Sharpe < 0.6×
backtest Sharpe over 100+ trades or other configurable thresholds.

**Effort**: ~150-200 LOC.

### Priority 5: Data versioning

Feature lake snapshots versioned by content hash. Backtest runs pin to
specific data versions for byte-identical reproducibility years later.
Old data versions retained per retention policy (default: last 4
versions, or all versions referenced by non-deprecated backtest runs).

**Effort**: ~200-300 LOC.

### Priority 6: Multi-model interaction backtest

Extension of integration backtest: runs multiple offensive models
together under portfolio risk cap. Reveals capital-competition
behavior invisible in isolation backtests.

**Effort**: ~200-300 LOC (extension of P2).

### Verification checklist

- [ ] Hybrid defensive model registered; backtest shows it beats rule-based-v1 and hmm-v1 baselines on classifier quality and forward-return-conditioning metrics
- [ ] Integration backtest reconstructs a historical session to configurable tolerance
- [ ] Walk-forward retrain produces a new version automatically on schedule; shadow → A/B → live promotion verified end-to-end
- [ ] Drift detection alerts on synthetic divergence test
- [ ] Data version pinning: re-running an old backtest run produces byte-identical metrics
- [ ] Multi-model integration backtest reveals capital competition cases that isolation backtests missed

## Open questions for v2.5 kickoff

1. **Model serialization format** — pickle is convenient but version-fragile. Consider joblib or a custom format that includes a manifest with Python / library versions. Decide before P2a starts.
2. **Backtest result retention policy** — full trade-level data is heavy. Default: keep full trades for last 10 runs per model + version; older runs keep aggregates only.
3. **Frontend framework decision (cross-references v2.4 P4)** — does the model registry surface justify moving to a component framework, or is HTMX morphing enough? Decide during v2.4 closeout.
4. **Feature lake ownership** — currently the engine consumes features inline. The roadmap assumes a separable feature lake (Parquet partitioned by date). Confirm this stages cleanly out of the existing data flow or whether it requires its own design pass.
5. **HMM "ground truth" for OOS validation** — what counts as a "correct" regime label historically? Options: forward-return-derived (use next-period return characteristics), expert-labeled (your own historical annotation), or unsupervised (no ground truth, evaluate only by economic utility). Decide before P1a v2.6.
